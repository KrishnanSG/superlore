Canvas

Author a whiteboard as a typed node-and-edge graph. Humans see an interactive FigJam-style board; agents read the same structure over the MCP — never a screenshot to interpret.

Component Dual-representation Copy for agent

Overview

A Canvas renders from a single fenced block. The same graph powers the visual board and the typed knowledge face the MCP serves — one source, two readers.

Why it matters

Typed, not pixels

Nodes, edges, intents and groups are real data — diff-able, queryable, and stable across renders.

One source, two readers

The board humans pan and zoom is the exact structure an agent reads — no drift, no interpretation.

Authoring

Tip

Start from layout flow and let ELK auto-arrange. Switch to board or manual only when you need authored placement.

1

Open a fenced block

Use a superlore-canvas code fence anywhere in your MDX.

2

Describe nodes & edges

Give each node a kind, intent, and label; connect them with typed edges.

3

Render & serve

Humans get the board; the MCP serves the same graph to agents — automatically.

Example

flow.mdx
```superlore-canvas
{
  "layout": "flow", "direction": "right",
  "nodes": [
    { "id": "src", "kind": "cylinder", "label": "Source" },
    { "id": "cv",  "kind": "rounded",  "intent": "accent" }
  ],
  "edges": [{ "from": "src", "to": "cv" }]
}
```
Interactive board — pan, zoom, inspect
The rendered Canvas. Locked or interactive; light and dark co-equal.
