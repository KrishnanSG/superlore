A `Timeline` is a vertical, dated rail ‚Äî each item is an event with a date, a title, an optional
body, a status dot, tags, and typed references. It's one of the **dual-representation** components:
it renders a polished rail for humans **and** hands an agent the parsed, range-queryable data
behind it ‚Äî never a picture to interpret.

<Note title="The shared knowledge envelope">
  Every structural knowledge face extends one shape ‚Äî `kind` discriminates the node, `id` is a
  stable slug addressable as `${pageId}#${id}`, and `refs[]` is the universal graph edge that powers
  `navigate`. Shared enums (`Status`, date precision) let a single query ‚Äî "everything
  `in-progress`", "everything dated in Q3" ‚Äî work across Timeline, EntityCard, and the rest with no
  per-component logic.
</Note>

## When to use it

Roadmaps, project histories, release sequences, incident timelines ‚Äî anything that is
fundamentally _ordered by time with state_. The agent gets parsed dates and statuses, never a
description of a rail.

## Props

**`Timeline`**

| Prop    | Type                  | Default      | Description                  |
| ------- | --------------------- | ------------ | ---------------------------- |
| `items` | `TimelineItemInput[]` | ‚Äî            | The events, in order         |
| `label` | `string`              | `"Timeline"` | Accessible name for the list |

**`TimelineItemInput`**

| Field    | Type                         | Default | Description                                                                                                |
| -------- | ---------------------------- | ------- | ---------------------------------------------------------------------------------------------------------- |
| `date`   | `string`                     | ‚Äî       | `2026`, `2026-Q3`, `2026-08`, or `2026-07-01` ‚Äî precision is inferred                                      |
| `title`  | `string`                     | ‚Äî       | The event title                                                                                            |
| `body`   | `ReactNode`                  | ‚Äî       | Optional detail under the title                                                                            |
| `status` | `Status`                     | ‚Äî       | One of `planned ¬∑ in-progress ¬∑ blocked ¬∑ done ¬∑ deprecated ¬∑ proposed ¬∑ accepted ¬∑ rejected ¬∑ superseded` |
| `tags`   | `string[]`                   | ‚Äî       | Chips rendered after the body                                                                              |
| `refs`   | `{ rel?, target, label? }[]` | ‚Äî       | Typed links rendered as pill-links                                                                         |

## What the agent sees

`get_component_data` returns the data, not the rail ‚Äî dates parsed to `{ iso, precision }`, the
status enum intact, and the ref resolved as internal:

```json
{
  "kind": "timeline",
  "id": "timeline",
  "items": [
    {
      "id": "project-kickoff",
      "date": { "iso": "2026-07-01", "precision": "day" },
      "title": "Project kickoff",
      "body": "Repo, brand, and thesis locked.",
      "status": "done"
    },
    {
      "id": "core-extraction",
      "date": { "iso": "2026-08", "precision": "month" },
      "title": "Core extraction",
      "body": "Components, theme, MCP.",
      "status": "in-progress",
      "tags": ["phase-2"]
    },
    {
      "id": "ga-launch",
      "date": { "iso": "2026-Q3", "precision": "quarter" },
      "title": "GA launch",
      "status": "planned",
      "tags": ["milestone"],
      "refs": [{ "rel": "related", "target": "/docs/agents/mcp", "label": "MCP", "internal": true }]
    }
  ]
}
```

<Tip title="Why precision matters">
  Because each date keeps its precision, the MCP can answer "what shipped in Q3?" ‚Äî matching the
  `quarter`-precision `2026-Q3` and any `2026-07`/`2026-08-12` that fall inside it ‚Äî without you
  normalizing dates by hand.
</Tip>
