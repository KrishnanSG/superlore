A `Diagram` is a theme-aware Mermaid diagram. It renders client-side with superlore-violet colours
and re-renders on theme change — and it serializes the **lossless Mermaid source** to the
knowledge face, so an agent receives the diagram definition, not a rendered SVG. `Mermaid` is an
alias.

## When to use it

A flow, sequence, state machine, or graph that is clearest as Mermaid. A parsed node/edge `graph`
extraction lands in a later phase; for an interactive, fully-structured graph today, reach for
[Canvas](/docs/canvas).

## Props

**`Diagram`** (alias `Mermaid`)

| Prop    | Type     | Default | Description                                        |
| ------- | -------- | ------- | -------------------------------------------------- |
| `chart` | `string` | —       | The Mermaid source (flowchart, sequence, state, …) |

## Live example

<Diagram
  chart={`flowchart LR
  Author -->|MDX| Index
  Index --> Humans
  Index --> Agents
`}
/>

## What the agent sees

The exact source, never a picture — `syntax` says how to read it:

```json
{
  "kind": "diagram",
  "id": "diagram",
  "syntax": "mermaid",
  "source": "flowchart LR\n  Author -->|MDX| Index\n  Index --> Humans\n  Index --> Agents"
}
```

## Diagram vs. Canvas

A `Diagram` and a `Canvas` share one knowledge shape — both are `diagram` kind. The difference is
the `syntax` and what's preserved: a Diagram carries the lossless Mermaid `source`; a Canvas
carries a typed node/edge `graph`, because its spec **is** the knowledge graph.

```json
{
  "kind": "diagram",
  "syntax": "canvas",
  "graph": {
    "nodes": [{ "id": "pg", "label": "Postgres", "kind": "cylinder" }],
    "edges": [{ "from": "api", "to": "pg", "rel": "related" }]
  }
}
```
